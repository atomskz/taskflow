Список задач для дальнейшего развития проекта. Формат рассчитан на выполнение AI-агентом: у каждой задачи есть цель, контекст, шаги, критерии готовности и затрагиваемые файлы.

## TASK-001: Серверные настройки пользователя

### Цель
Хранить пользовательские настройки (тема, первый день недели, показ выполненных, число задач на дашборде) на сервере, а не в `localStorage`, чтобы они синхронизировались между устройствами.

### Контекст
Сейчас `settings` живут только в браузере (`client/src/store.jsx`, ключ `taskflow.settings.v1`). Это UI-предпочтения, но синхронизация между устройствами требует хранения в БД.

### Что нужно сделать
- Добавить таблицу `user_settings` (или JSON-колонку `settings` в `users`).
- Endpoints: `GET /api/settings`, `PATCH /api/settings`.
- В `client/src/api/` добавить `settings.js`; в `store.jsx` загружать настройки после входа и сохранять через API (с дебаунсом).
- Сохранить локальный кэш как fallback до загрузки.

### Критерии готовности
- Изменение настроек сохраняется на сервере и переживает вход с другого браузера.
- Невалидные значения отклоняются (zod).
- Есть тест на чтение/обновление настроек.

### Затрагиваемые файлы
- `server/src/db/schema.sql`, `server/src/modules/settings/*` (новый модуль)
- `server/src/app.js`
- `client/src/api/settings.js` (новый), `client/src/store.jsx`

## TASK-002: Тёмная тема

### Цель
Реализовать рабочую тёмную тему вместо текущей заглушки.

### Контекст
В `client/src/pages/SettingsPage.jsx` кнопка тёмной темы показывает toast «появится в версии 2». Дизайн-токены вынесены в CSS-переменные (`client/src/index.css`, `:root`).

### Что нужно сделать
- Определить набор переменных для тёмной темы (`:root[data-theme="dark"]` или класс на `<html>`).
- Применять `settings.theme` к корневому элементу в `store.jsx`/`main.jsx`.
- Включить реальное переключение в `SettingsPage.jsx` (убрать заглушку `setDark`).

### Критерии готовности
- Переключение светлая/тёмная мгновенно меняет интерфейс без перезагрузки.
- Выбор сохраняется (см. TASK-001 или localStorage).
- Нет «выпавших» из темы элементов (графики, модалки, тосты).

### Затрагиваемые файлы

## TASK-003: Серверная фильтрация, сортировка и пагинация задач

### Цель
Перенести фильтрацию/сортировку/поиск и пагинацию на сервер для масштабирования (комфортная работа с сотнями–тысячами задач).

### Контекст
Сейчас `GET /api/tasks` возвращает все задачи пользователя, а `filterSortTasks` (`client/src/lib/tasks.js`) фильтрует на клиенте. ТЗ требует комфортной работы с 500 задачами (раздел 19.1).

### Что нужно сделать
- Расширить `GET /api/tasks` query-параметрами: `status`, `priority`, `search`, `onlyOverdue`, `onlyToday`, `sort`, `limit`, `offset`.
- Реализовать фильтрацию в `tasks.repo.js`/`tasks.service.js` (параметризованный SQL).
- Возвращать `{ tasks, total }`.
- В `store.jsx`/`api/tasks.js` передавать фильтры; дашборд может получать агрегаты отдельным endpoint (см. TASK-004).

### Критерии готовности
- Фильтры/поиск/сортировка работают через сервер и совпадают по поведению с текущим клиентским вариантом.
- Пагинация работает; UI подгружает страницы.
- Тесты на ключевые комбинации фильтров.

### Затрагиваемые файлы
- `server/src/modules/tasks/*`, `client/src/api/tasks.js`, `client/src/store.jsx`, `client/src/lib/tasks.js`, `client/src/pages/TasksPage.jsx`

## TASK-004: Endpoint агрегатов дашборда

### Цель
Считать статистику дашборда на сервере, чтобы не тянуть весь список задач ради цифр.

### Контекст
`buildDashboard` (`client/src/lib/tasks.js`) агрегирует на клиенте. При серверной пагинации (TASK-003) клиент уже не будет иметь полный список.

### Что нужно сделать
- `GET /api/dashboard` → счётчики (активные, на сегодня, просрочено, за неделю, высокий приоритет), распределения по статусам/приоритетам, прогресс недели, предстоящие/просроченные/текущие.
- Реализовать SQL-агрегации в новом сервисе.
- В `DashboardPage.jsx` потреблять endpoint; оставить клиентский расчёт как fallback или убрать.

### Критерии готовности
- Цифры дашборда совпадают с текущими при тех же данных.
- Один-два запроса вместо загрузки всех задач.
- Тест на корректность агрегатов на фикстуре.

### Затрагиваемые файлы
- `server/src/modules/dashboard/*` (новый), `server/src/app.js`, `client/src/api/`, `client/src/pages/DashboardPage.jsx`

## TASK-005: Refresh-токены и httpOnly-cookie

### Цель
Повысить безопасность сессий: короткоживущий access-токен + refresh-токен, хранение в httpOnly-cookie вместо `localStorage`.

### Контекст
Сейчас JWT (7 дней) хранится в `localStorage` (`client/src/api/client.js`), серверного отзыва нет.

### Что нужно сделать
- Access-токен ~15 мин, refresh-токен в httpOnly+Secure cookie; таблица/хранилище refresh-токенов с возможностью отзыва.
- Endpoints: `POST /api/auth/refresh`, доработать `logout` (отзыв refresh).
- На клиенте — авто-рефреш при `401` в `client.js`.

### Критерии готовности
- Истёкший access-токен прозрачно обновляется.
- Logout отзывает refresh-токен на сервере.
- Тесты на refresh/отзыв.

### Затрагиваемые файлы
- `server/src/modules/auth/*`, `server/src/utils/token.js`, `server/src/db/schema.sql`, `client/src/api/client.js`, `client/src/api/auth.js`

## TASK-006: Мягкое удаление и корзина

### Цель
Заменить физическое удаление на мягкое (`deleted_at`) с возможностью восстановления.

### Контекст
ТЗ (6.4.4) допускает мягкое удаление для расширенной версии. Сейчас `DELETE /api/tasks/:id` удаляет строку физически.

### Что нужно сделать

- `DELETE` ставит `deleted_at`; endpoint восстановления; (опц.) UI «Корзина».

### Критерии готовности

- Тесты на удаление/восстановление.

### Затрагиваемые файлы

## TASK-007: Hardening и production-готовность бэкенда

### Цель
Подготовить API к публичному деплою.

### Контекст
Сейчас нет rate limiting, security-заголовков, structured-логов; `JWT_SECRET` имеет дев-дефолт.

### Что нужно сделать

- Dockerfile/инструкция деплоя; раздача собранного фронтенда (или отдельный хостинг).

### Критерии готовности
- Security-заголовки присутствуют; брутфорс логина ограничен.
- Прод не стартует с небезопасным секретом.
- Документация деплоя обновлена.
