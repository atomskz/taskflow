# Техническое задание

## Проект: TODO List с авторизацией, дашбордом и календарём

## 1. Общее описание проекта

### 1.1. Название проекта

**TaskFlow** — веб-приложение для управления личными задачами с авторизацией пользователя, дашбордом, календарём и аналитикой по задачам.

Название можно заменить на любое другое: `MyTodo`, `TaskBoard`, `Todo Calendar`, `FocusList`.

---

## 2. Цель проекта

Создать учебный, но приближенный к реальному веб-проект, который позволит пользователю:

* зарегистрироваться и войти в личный аккаунт;
* создавать, редактировать, удалять и завершать задачи;
* прикреплять задачи к дате и времени в календаре;
* видеть список текущих, просроченных и предстоящих задач;
* отслеживать прогресс через дашборд и инфографику;
* фильтровать и сортировать задачи по статусу, приоритету и срокам.

Проект должен продемонстрировать навыки frontend-разработчика:

* работа с компонентной архитектурой;
* маршрутизация;
* формы и валидация;
* авторизация;
* работа с API или backend-as-a-service;
* хранение пользовательских данных;
* работа с датами;
* адаптивная верстка;
* состояние приложения;
* обработка ошибок;
* базовые тесты;
* деплой.

---

## 3. Целевая аудитория

Приложение предназначено для индивидуального пользователя, который хочет вести личный список задач.

Примеры пользователей:

* начинающий разработчик, который планирует учебные задачи;
* студент, который отслеживает дедлайны;
* фрилансер, который ведёт рабочие задачи;
* обычный пользователь, которому нужен простой календарный TODO-лист.

---

## 4. Основные сущности проекта

### 4.1. Пользователь

Пользователь — человек, который зарегистрирован в системе и имеет личный список задач.

Поля пользователя:

```ts
type User = {
  id: string;
  name: string;
  email: string;
  avatarUrl?: string;
  createdAt: string;
  updatedAt: string;
};
```

### 4.2. Задача

Задача — основная сущность приложения.

Поля задачи:

```ts
type Task = {
  id: string;
  userId: string;

  title: string;
  description?: string;

  status: 'todo' | 'in_progress' | 'done' | 'archived';
  priority: 'low' | 'medium' | 'high' | 'critical';

  dueDate?: string;
  startDate?: string;
  completedAt?: string;

  calendarDate?: string;
  calendarStartTime?: string;
  calendarEndTime?: string;

  tags?: string[];

  createdAt: string;
  updatedAt: string;
};
```

### 4.3. Статусы задач

В проекте должны быть следующие статусы:

| Статус        | Описание                                  |
| ------------- | ----------------------------------------- |
| `todo`        | Задача создана, но работа ещё не началась |
| `in_progress` | Задача находится в работе                 |
| `done`        | Задача выполнена                          |
| `archived`    | Задача скрыта из основного списка         |

### 4.4. Приоритеты задач

| Приоритет  | Описание            |
| ---------- | ------------------- |
| `low`      | Низкий приоритет    |
| `medium`   | Обычный приоритет   |
| `high`     | Высокий приоритет   |
| `critical` | Критичный приоритет |

---

## 5. Роли пользователей

В MVP используется только одна роль:

### Обычный пользователь

Может:

* регистрироваться;
* входить в аккаунт;
* выходить из аккаунта;
* создавать задачи;
* редактировать свои задачи;
* удалять свои задачи;
* менять статус задач;
* смотреть дашборд;
* смотреть календарь;
* управлять только своими задачами.

Администраторская роль в MVP не требуется.

---

## 6. Функциональные требования

# 6.1. Авторизация

## 6.1.1. Регистрация

Пользователь должен иметь возможность создать аккаунт.

Форма регистрации должна содержать:

* имя;
* email;
* пароль;
* подтверждение пароля.

### Валидация регистрации

Имя:

* обязательное поле;
* минимум 2 символа;
* максимум 50 символов.

Email:

* обязательное поле;
* должен быть валидным email-адресом;
* не должен уже использоваться другим пользователем.

Пароль:

* обязательное поле;
* минимум 8 символов;
* должен содержать хотя бы одну букву;
* должен содержать хотя бы одну цифру.

Подтверждение пароля:

* обязательное поле;
* должно совпадать с паролем.

### Поведение после регистрации

После успешной регистрации пользователь должен быть:

* либо автоматически авторизован;
* либо перенаправлен на страницу входа.

Для MVP рекомендуется автоматическая авторизация после регистрации.

---

## 6.1.2. Вход

Пользователь должен иметь возможность войти в аккаунт.

Форма входа должна содержать:

* email;
* пароль.

### Валидация входа

Email:

* обязательное поле;
* должен быть валидным email-адресом.

Пароль:

* обязательное поле.

### Ошибки входа

Приложение должно показывать понятные ошибки:

* пользователь не найден;
* неверный пароль;
* проблема соединения;
* неизвестная ошибка.

Не рекомендуется показывать слишком технические сообщения пользователю.

Плохо:

```text
AuthError: invalid credentials code 401
```

Хорошо:

```text
Неверный email или пароль.
```

---

## 6.1.3. Выход

Пользователь должен иметь возможность выйти из аккаунта.

После выхода:

* токен авторизации должен быть удалён;
* пользователь должен быть перенаправлен на страницу входа;
* защищённые страницы должны стать недоступны.

---

## 6.1.4. Защищённые маршруты

Следующие страницы должны быть доступны только авторизованному пользователю:

* дашборд;
* список задач;
* календарь;
* создание задачи;
* редактирование задачи;
* профиль;
* настройки.

Если неавторизованный пользователь пытается открыть защищённую страницу, его нужно перенаправить на `/login`.

---

# 6.2. Главная страница

Главная страница доступна всем пользователям.

Она должна содержать:

* краткое описание проекта;
* кнопку “Войти”;
* кнопку “Зарегистрироваться”;
* краткий список возможностей:

  * управление задачами;
  * календарь;
  * дашборд;
  * статистика;
  * приоритеты.

Для авторизованного пользователя главная страница может показывать кнопку “Перейти в дашборд”.

---

# 6.3. Дашборд

Дашборд — главная страница авторизованного пользователя.

Он должен давать краткую сводку по задачам.

## 6.3.1. Виджеты дашборда

На дашборде должны быть следующие блоки:

### 1. Сводка по задачам

Карточки со значениями:

* всего активных задач;
* задач на сегодня;
* просроченных задач;
* выполненных задач за неделю;
* задач с высоким приоритетом.

Пример:

```text
Активные задачи: 12
На сегодня: 3
Просрочено: 2
Выполнено за неделю: 8
Высокий приоритет: 4
```

### 2. Предстоящие задачи

Список ближайших задач.

Правила отображения:

* показывать задачи на ближайшие 7 дней;
* сортировать по дате;
* сначала показывать более срочные задачи;
* максимум 5–7 задач на дашборде;
* рядом показывать дату, время и приоритет.

### 3. Текущие задачи

Список задач со статусом:

* `todo`;
* `in_progress`.

Задачи со статусом `done` не должны попадать в текущие.

### 4. Просроченные задачи

Отдельный блок для задач, у которых:

* `dueDate` меньше текущей даты;
* статус не `done`;
* статус не `archived`.

### 5. Инфографика по статусам

Диаграмма, показывающая распределение задач:

* todo;
* in_progress;
* done;
* archived.

Например:

```text
Todo: 5
In progress: 3
Done: 10
Archived: 2
```

### 6. Инфографика по приоритетам

Диаграмма, показывающая количество задач по приоритетам:

* low;
* medium;
* high;
* critical.

### 7. Прогресс недели

Блок, показывающий:

* сколько задач было создано за неделю;
* сколько задач было выполнено за неделю;
* процент выполнения.

Пример формулы:

```text
Процент выполнения = выполненные задачи / созданные задачи * 100
```

Если за неделю не было создано задач, процент выполнения можно считать равным 0.

---

# 6.4. Список задач

Страница задач должна показывать все задачи пользователя.

## 6.4.1. Основной список

Каждая задача в списке должна отображать:

* название;
* описание, если есть;
* статус;
* приоритет;
* дату дедлайна;
* дату и время в календаре, если задача прикреплена к календарю;
* теги;
* кнопки действий:

  * открыть;
  * редактировать;
  * завершить;
  * удалить.

---

## 6.4.2. Создание задачи

Пользователь должен иметь возможность создать новую задачу.

Форма создания задачи должна содержать:

| Поле             | Тип                  | Обязательное |
| ---------------- | -------------------- | ------------ |
| Название         | text input           | Да           |
| Описание         | textarea             | Нет          |
| Статус           | select               | Да           |
| Приоритет        | select               | Да           |
| Дата дедлайна    | date picker          | Нет          |
| Дата в календаре | date picker          | Нет          |
| Время начала     | time picker          | Нет          |
| Время окончания  | time picker          | Нет          |
| Теги             | input / multi-select | Нет          |

### Валидация создания задачи

Название:

* обязательное поле;
* минимум 3 символа;
* максимум 100 символов.

Описание:

* максимум 1000 символов.

Статус:

* по умолчанию `todo`.

Приоритет:

* по умолчанию `medium`.

Дата дедлайна:

* может быть пустой;
* если указана, должна быть валидной датой.

Время окончания:

* не может быть раньше времени начала.

---

## 6.4.3. Редактирование задачи

Пользователь должен иметь возможность изменить:

* название;
* описание;
* статус;
* приоритет;
* дату дедлайна;
* дату календаря;
* время начала;
* время окончания;
* теги.

После сохранения изменений:

* задача должна обновиться в списке;
* данные на дашборде должны пересчитаться;
* календарь должен отобразить актуальную информацию.

---

## 6.4.4. Удаление задачи

Пользователь должен иметь возможность удалить задачу.

Перед удалением нужно показать подтверждение:

```text
Вы уверены, что хотите удалить задачу?
```

После подтверждения задача удаляется.

Для MVP допустимо физическое удаление задачи.

Для более продвинутой версии лучше использовать мягкое удаление:

```ts
deletedAt: string | null;
```

---

## 6.4.5. Завершение задачи

Пользователь должен иметь возможность быстро отметить задачу как выполненную.

После завершения:

* статус задачи становится `done`;
* заполняется поле `completedAt`;
* задача исчезает из блока текущих задач;
* задача начинает учитываться в статистике выполненных задач.

---

## 6.4.6. Архивация задачи

Архивация — необязательная функция для MVP, но полезная для расширенной версии.

Архивная задача:

* не отображается в основном списке;
* не отображается на дашборде как активная;
* может быть найдена через фильтр “Архив”.

---

# 6.5. Фильтрация и сортировка задач

## 6.5.1. Фильтры

На странице задач должны быть фильтры:

* по статусу;
* по приоритету;
* по дате;
* по тегу;
* по наличию дедлайна;
* по просроченности.

Минимальный набор для MVP:

* фильтр по статусу;
* фильтр по приоритету;
* фильтр “только просроченные”;
* фильтр “на сегодня”.

---

## 6.5.2. Поиск

Пользователь должен иметь возможность искать задачи по:

* названию;
* описанию;
* тегам.

Поиск должен работать без перезагрузки страницы.

---

## 6.5.3. Сортировка

Пользователь должен иметь возможность сортировать задачи:

* по дате создания;
* по дедлайну;
* по приоритету;
* по статусу;
* по алфавиту.

Для MVP достаточно:

* новые сверху;
* ближайший дедлайн сверху;
* высокий приоритет сверху.

---

# 6.6. Календарь

Календарь нужен для отображения задач, привязанных к дате.

## 6.6.1. Основной календарь

Календарь должен иметь минимум один режим отображения:

* месяц.

Расширенные режимы:

* неделя;
* день;
* список.

Для MVP достаточно режима “месяц”.

---

## 6.6.2. Отображение задач в календаре

Если у задачи заполнено поле `calendarDate`, она должна отображаться в календаре.

В ячейке дня нужно показывать:

* название задачи;
* цвет или маркер приоритета;
* статус задачи.

Если задач в одном дне много:

* показывать первые 2–3 задачи;
* остальные скрывать под ссылкой “ещё N”.

Пример:

```text
24 июня
- Сделать форму логина
- Прочитать документацию React
+ ещё 3
```

---

## 6.6.3. Создание задачи из календаря

Пользователь должен иметь возможность нажать на дату в календаре и создать задачу на эту дату.

Поведение:

1. Пользователь нажимает на дату.
2. Открывается форма создания задачи.
3. Поле `calendarDate` уже заполнено выбранной датой.
4. Пользователь заполняет остальные поля.
5. После сохранения задача появляется в календаре.

---

## 6.6.4. Редактирование задачи из календаря

Пользователь должен иметь возможность нажать на задачу в календаре.

После нажатия открывается:

* модальное окно;
* либо отдельная страница задачи.

В окне нужно показать:

* название;
* описание;
* статус;
* приоритет;
* дата;
* время;
* кнопка редактирования;
* кнопка завершения;
* кнопка удаления.

---

## 6.6.5. Связь дедлайна и календарной даты

В проекте нужно разделять два понятия:

### Дедлайн

Дата, до которой задача должна быть выполнена.

Поле:

```ts
dueDate
```

### Дата в календаре

Дата, на которую пользователь запланировал выполнение задачи.

Поле:

```ts
calendarDate
```

Пример:

```text
Задача: Подготовить отчёт
Дата в календаре: 20 июня
Дедлайн: 25 июня
```

Это значит, что пользователь планирует работать над задачей 20 июня, но крайний срок — 25 июня.

Для MVP можно упростить и использовать только одну дату, но лучше сразу заложить разделение.

---

# 6.7. Страница задачи

У каждой задачи должна быть отдельная страница или подробное модальное окно.

На странице задачи отображается:

* название;
* описание;
* статус;
* приоритет;
* дата создания;
* дата обновления;
* дедлайн;
* дата в календаре;
* время начала;
* время окончания;
* теги;
* кнопка редактирования;
* кнопка удаления;
* кнопка завершения.

---

# 6.8. Профиль пользователя

Страница профиля должна содержать:

* имя пользователя;
* email;
* дата регистрации;
* количество созданных задач;
* количество выполненных задач.

Для MVP редактирование профиля можно не делать.

Расширенная версия:

* изменение имени;
* загрузка аватара;
* изменение пароля;
* удаление аккаунта.

---

# 6.9. Настройки

Страница настроек необязательна для MVP, но может быть полезна.

Возможные настройки:

* светлая/тёмная тема;
* формат даты;
* первый день недели: понедельник/воскресенье;
* показывать или скрывать завершённые задачи;
* количество задач на дашборде.

Для первого проекта рекомендуется реализовать только переключение темы.

На странице должно быть:

* сообщение “Страница не найдена”;
* кнопка возврата на главную;
* если пользователь авторизован — кнопка перехода в дашборд.

## 8. UI/UX требования

## 8.1. Общий стиль

Интерфейс должен быть:

* чистым;
* минималистичным;
* понятным;
* адаптивным;
* без перегрузки визуальными элементами.

## 12. Бизнес-логика

## 12.1. Активная задача

Активной считается задача, у которой статус:

* `todo`;
* `in_progress`.

Задачи со статусами `done` и `archived` не считаются активными.

---

## 12.2. Просроченная задача

Задача считается просроченной, если:

```text
dueDate < today
AND status !== 'done'
AND status !== 'archived'
```

---

## 12.3. Задача на сегодня

Задача считается задачей на сегодня, если:

```text
calendarDate === today
OR dueDate === today
```

---

## 12.4. Предстоящая задача

Задача считается предстоящей, если:

```text
date >= today
AND date <= today + 7 days
AND status !== 'done'
AND status !== 'archived'
```

Где `date` — это `calendarDate` или `dueDate`.

---

## 12.5. Выполненная задача за неделю

Задача считается выполненной за неделю, если:

```text
completedAt >= startOfWeek
AND completedAt <= endOfWeek
AND status === 'done'
```

## 22. Пользовательские сценарии

## 22.1. Регистрация нового пользователя

1. Пользователь открывает сайт.
2. Нажимает “Зарегистрироваться”.
3. Вводит имя, email и пароль.
4. Подтверждает пароль.
5. Нажимает “Создать аккаунт”.
6. Система создаёт аккаунт.
7. Пользователь попадает на дашборд.

Критерий приёмки:

```text
Пользователь с валидными данными может создать аккаунт и попасть в личный кабинет.
```

---

## 22.2. Вход пользователя

1. Пользователь открывает страницу входа.
2. Вводит email и пароль.
3. Нажимает “Войти”.
4. Система проверяет данные.
5. Пользователь попадает на дашборд.

Критерий приёмки:

```text
Пользователь с корректными данными может войти в систему.
```

---

## 22.3. Создание задачи

1. Пользователь открывает страницу задач.
2. Нажимает “Создать задачу”.
3. Заполняет название.
4. Выбирает приоритет.
5. Указывает дедлайн.
6. Нажимает “Сохранить”.
7. Задача появляется в списке.

Критерий приёмки:

```text
После создания задача отображается в списке задач и учитывается на дашборде.
```

---

## 22.4. Создание задачи из календаря

1. Пользователь открывает календарь.
2. Нажимает на дату.
3. Открывается форма создания задачи.
4. Дата календаря уже заполнена.
5. Пользователь вводит название задачи.
6. Сохраняет задачу.
7. Задача отображается в выбранной дате календаря.

Критерий приёмки:

```text
Задача, созданная из календаря, автоматически получает выбранную дату.
```

---

## 22.5. Завершение задачи

1. Пользователь видит задачу в списке.
2. Нажимает “Завершить”.
3. Статус задачи меняется на `done`.
4. Задача перестаёт считаться активной.
5. Статистика на дашборде обновляется.

Критерий приёмки:

```text
Выполненная задача не отображается как активная и учитывается в статистике выполненных задач.
```

---

## 22.6. Просмотр просроченных задач

1. Пользователь открывает дашборд.
2. Система проверяет задачи с дедлайнами.
3. Просроченные задачи отображаются в отдельном блоке.

Критерий приёмки:

```text
Задачи с прошедшим дедлайном и незавершённым статусом отображаются как просроченные.
```

## 30. Итоговое определение MVP

MVP проекта — это веб-приложение, в котором пользователь может зарегистрироваться, войти, создать свои задачи, назначить им дату, увидеть их в списке, на дашборде и в календаре, а также завершать, редактировать и удалять задачи.

MVP считается завершённым, если приложение можно показать другому человеку, и он без объяснений сможет:

* создать аккаунт;
* войти;
* создать задачу;
* назначить дату;
* увидеть задачу в календаре;
* завершить задачу;
* посмотреть статистику на дашборде.
